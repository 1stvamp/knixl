# System assembly flake emission design (#40)

Date: 2026-07-22
Status: approved, ready for implementation plan
Issue: #40
ADR: 0009 (system assembly flake emission)

Let `knixl generate` emit a bootable system: when `knixl.kdl` declares a `system {}` block,
generate writes `generated/flake.nix` defining `nixosConfigurations.<host>` for every host,
pinned per host to its baseline nixpkgs rev, as a generated and locked artefact. Absent the
block, knixl emits modules only, exactly as today.

## Grounding (current state)

- `knixl.kdl` is the project-root config, parsed by `knixl_pipeline::project::parse_project`
  into `ProjectConfig { default_release, oracle_modules }` (an absent file is
  `ProjectConfig::default()`). This is where `system {}` lands.
- Hosts declare `system "x86_64-linux"` (the Nix system double, lowered to
  `nixpkgs.hostPlatform` in the generated module) and `nixpkgs release="<rel>"` (the baseline,
  resolved to a rev per host and stored on the lock's `baseline` line, ADR 0007).
- `knixl generate` builds a `BTreeMap<PathBuf, String>` of generated files (host modules under
  `generated/hosts/<h>.nix`) via the pipeline; `Plan::compute` reconciles each against the lock
  (`Stale`/`Missing`/`Drifted`/`Orphaned`), and generated text is formatted by the pinned
  formatter and hashed post-format. The generated flake joins this same map and machinery.
- A full 40-char rev is a pure pin (ADR 0005), so a flake can `builtins.fetchGit { url; rev; }`
  nixpkgs with no `flake.lock`. `knixl.lock.kdl` stays the single lock.
- `output = f(kdl, tool_version, module_versions, formatter_version, oracle_rev, baselines)`.

## Design

### 1. Opt-in: the `system {}` block in `knixl.kdl`

Extend `ProjectConfig` with `system: Option<SystemConfig>` where
`SystemConfig { state_version: String, nixpkgs_url: String }`. `parse_project` reads a
top-level `system` node:

```kdl
system {
    state-version "25.05"
    // optional: nixpkgs-url "https://github.com/NixOS/nixpkgs" (default)
}
```

- `state-version` is required inside `system {}` (it has no other source and `nixosSystem`
  needs `system.stateVersion`); a `system {}` block without it is a parse/validation error.
- `nixpkgs-url` is optional, default `https://github.com/NixOS/nixpkgs`, so the flake's
  `fetchGit` url is configurable for a fork or mirror.
- `system` absent (or no `system` node) yields `system: None` and today's behaviour.

### 2. The emitter (pure, in knixl-pipeline)

A new `knixl_pipeline::flake` module with a pure function:

```rust
pub struct FlakeHost {
    pub name: String,          // nixosConfigurations key
    pub baseline_rev: String,  // resolved per-host nixpkgs rev (ADR 0007)
    pub module_path: String,   // path relative to generated/, e.g. "./hosts/nas.nix"
}

pub fn render_system_flake(
    hosts: &[FlakeHost],       // sorted by name by the caller for determinism
    state_version: &str,
    nixpkgs_url: &str,
) -> String;
```

Emitted shape (pre-format; the pinned formatter normalises layout, so only structure matters,
and the post-format text is what is hashed):

```nix
# Generated by knixl. Do NOT edit; regenerate with `knixl generate`.
{
  description = "knixl-generated system";

  outputs = { ... }:
    let
      pkgsAt = rev: import (builtins.fetchGit {
        url = "<nixpkgs-url>";
        rev = rev;
      }) { };
    in
    {
      nixosConfigurations = {
        "<host>" = (pkgsAt "<baseline-rev>").lib.nixosSystem {
          modules = [
            ./hosts/<host>.nix
            { system.stateVersion = "<state-version>"; }
          ];
        };
        # ... one per host, in host-name order ...
      };
    };
}
```

- No `inputs`: the flake is self-contained and pure (`fetchGit` with a full rev is pure).
- `pkgsAt rev` fetches nixpkgs at a host's baseline rev and uses its `lib.nixosSystem`, so each
  host is built against its own baseline (ADR 0007). Architecture comes from the host module's
  `nixpkgs.hostPlatform`.
- `system.stateVersion` is injected as an inline module from `state-version`.
- Determinism: hosts are emitted in `name` order; the function is pure and byte-stable for a
  given input. No `HashMap` on the path.
- The flake composes only `./hosts/<host>.nix`. When a host later gains disko (#37) or secrets
  (#38) modules, those are part of that host's own generated output and are imported by the
  host module, not wired here.

### 3. Pipeline integration (generate / plan / lock / drift)

- When `ProjectConfig.system` is `Some`, the generate step adds `generated/flake.nix` to the
  generated-files map with the emitted-then-formatted text. It then flows through the exact
  same `Plan::compute` reconciliation, formatter, and lock hashing as host modules: it is
  `Stale`/`Missing` on input change, `Drifted` when hand-edited (refused without
  `--accept-drift`), and `Orphaned` (removed under `--prune`) when the `system {}` block is
  deleted.
- When `system` is `None`, no flake entry is produced and nothing about today's behaviour
  changes.

### 4. Baseline precondition

Emitting a host's `nixosConfigurations` entry requires that host's baseline rev, which lives on
the lock's per-host `baseline` line (ADR 0007). With `system {}` enabled:

- A host with a resolved baseline rev emits normally.
- A host with no resolved baseline (never `install`/`upgrade`d a baseline, or declared none)
  cannot be pinned. `generate` refuses with exit 5 and a message naming the host and pointing at
  `install`/`upgrade`, mirroring how an unresolved package pin refuses. This keeps the flake
  honest: it never emits an unpinned or defaulted nixpkgs.

A host with no `nixpkgs release` declared at all: `system {}` requires each host to have a
resolved baseline, so a host lacking a declared release is refused the same way (there is
nothing to pin). This is stated in the error.

### 5. Consumption and docs

- `nixos-rebuild switch --flake .#<host>` and `nixos-anywhere --flake .#<host>` consume the
  generated flake directly.
- docs/05 (CLI) documents that `generate` emits `generated/flake.nix` only when `system {}` is
  declared, and docs/01 (architecture) notes the flake as a generated artefact. A short note
  states plainly: without `system {}`, `knixl generate` produces modules, not a bootable system
  (the hand-written seam), per ADR 0009.

## Testing

Library (`knixl-pipeline`):
- `render_system_flake` is deterministic (same input, byte-identical output) and emits hosts in
  name order regardless of input order.
- The emitted flake, for a two-host input, contains `nixosConfigurations."<a>"` and
  `"<b>"`, each `pkgsAt "<rev>"` with that host's rev, `./hosts/<h>.nix`, and the
  `system.stateVersion` inline module; and the emitted text is valid Nix, validated by running
  it through the pinned formatter in an integration test.
- `parse_project` reads a `system { state-version "25.05" }` block into
  `ProjectConfig.system`, defaults `nixpkgs-url`, errors on a `system {}` block missing
  `state-version`, and yields `None` when the block is absent.
- Generate integration: a fixture project with `system {}` and resolved baselines produces a
  `generated/flake.nix` entry that reconciles clean and hashes into the lock; the same project
  with a host missing its baseline refuses with exit 5; a project without `system {}` produces
  no flake entry (unchanged behaviour).
- A golden test: a fixture flake under `examples/expected/flake.nix` reproduced byte-for-byte
  through the pinned formatter, matching the golden-test pattern the host modules already use.

## Out of scope

- disko (#37) and secrets (#38): their own modules; the flake only composes whatever a host
  generates.
- Generating `hardware-configuration.nix` from KDL, or importing an external hand-written one
  via a bespoke node (a host uses `raw-nix` if it must).
- Multi-architecture cross-build, non-NixOS targets, home-manager configurations, and any deploy
  orchestration beyond emitting `nixosConfigurations`.
- Flake inputs / `flake.lock` participation: knixl owns the lock (ADR 0009).
