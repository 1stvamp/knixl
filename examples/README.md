# examples: the behaviour contract

These are golden tests, not decoration. `hosts/*.kdl` are inputs. `expected/*.nix` is what
the pipeline must reproduce (post-nixfmt). `knixl.lock.kdl` is the lock shape.

Wire them early (see CLAUDE.md): parse the inputs, generate, format, compare byte-for-byte
against `expected/`, and diff the produced lock against `knixl.lock.kdl`.

Notes:
- Hashes in `knixl.lock.kdl` are real, not placeholders: the golden test suite
  (`cargo test -p knixl-pipeline --test golden`) regenerates every host and checks the output,
  the lock's structural round-trip, and pin pruning. All pass, none ignored. The generated
  header line is `# Generated by knixl <version> from <input>` (see `expected/*.nix`).
- `db` exercises multi-file output (host imports db-backup.nix), a runtime `lib.mkIf` from the
  `backups` module (Rust-only condition), and a conditional `lib.mkForce` from `postgres`.
  `backups` is a fully implemented, tested built-in module (id `backups`, v0.2.1).
- `web` exercises the declarative `web-service` module, `when-flag` (hardened), and the
  `raw-nix` escape hatch.
- `shared` exercises let-hoisting: the same `security-headers` block applied to two vhosts
  is bound once and referenced twice, plus a per-host `nixpkgs release` baseline.
- `pinned` and `pinned-override` exercise package version pinning: a pinned package alongside
  an ambient one, contrasting the `commit-mix` strategy (default) against `override`
  (`pkgs.<pkg>.overrideAttrs` against the historical pin).
- Running these examples directly needs the repo-root `modules/` directory alongside them:
  the declarative modules `web-service` and `security-headers` live there, and knixl discovers
  `modules/` as a sibling of `hosts/`. From a repo checkout both are already present; if
  copying `examples/` elsewhere, copy `modules/` too.
